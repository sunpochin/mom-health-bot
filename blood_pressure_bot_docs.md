# Susi 媽媽血壓記錄 LINE Bot 系統說明文件

本文件詳細記載了 Susi 媽媽血壓記錄 LINE Bot 的核心邏輯、客製化判定標準，以及曾遇到的技術問題（如 GAS 時區 Bug）與解決方案，方便日後維護與參考。

## 1. 客製化血壓警示標準 (長輩專屬)

針對長輩（82歲、有 1.4 cm 未破裂腦動脈瘤、骨鬆、曾有低血壓病史）的身體狀況，血壓的控制原則是：**「防範高壓衝高（避免動脈瘤破裂），同時防範低壓過低（避免頭暈跌倒）；但對舒張壓稍低（55-59 mmHg）降噪，避免警報疲勞。」**

## 2. 時間戳記與時區修正 (防踩雷經驗)

為了讓記錄與歷史清單能精確核對量測時間，系統進行了時間戳記的擴充，並解決了 Google Apps Script 經典的時區偏離 Bug。

### 🕒 當天即時登載補全
*   當 Susi 送出即時血壓（例如 `109 / 52`），系統寫入試算表第一欄的日期會從原本的 `6/29` 自動補上當前台灣時間，存為 `"6/29 16:34"` 字串。
*   LINE 的通知訊息也會顯示當次紀錄的時間點。

### ⚠️ GAS 時區 Bug 與解決方案
> [!IMPORTANT]
> **為什麼歷史紀錄曾出現明天的日期（例如 `6/30`）？**
> *   **原因**：當文字 `"6/29 16:34"` 寫入試算表後，會被 Google Sheet 自動判定為日期格式。當我們在 GAS 中使用 `getValues()` 讀取時，GAS 會將其轉為專案時區的 `Date` 物件。如果 GAS 專案時區與試算表時區不同（相差 15 小時），該物件對應的時間點便會偏移，轉換為字串時即會多加 15 小時變成 `6/30 07:34`。
> *   **解決方法**：歷史紀錄讀取 (`getRecentSummary`) 與重複防呆比對 (`isDuplicateEntry`) **一律改用 `getDisplayValues()`**。這會直接讀取儲存格上「顯示的純文字」（例如 `"6/29 16:34"`），完全繞過了 `Date` 物件的時區解析，從根本上解決了時差漂移問題。

### ☀️/🌙 時段自動判定
為確保不論伺服器時區如何漂移，早晚時段皆能 100% 判定正確，時段自動判定邏輯改為直接解析 `getTaipeiTimeString()` 拿到的台北時間字串（例如 `"16:34"`），取出小時數字比對：
*   **小時 $\ge$ 12**：判定為 `晚 (Malam / 🌙)`。
*   **小時 < 12**：判定為 `早 (Pagi / ☀️)`。

## 3. 最近歷史紀錄顯示 (最近四筆)

為讓長輩與照顧者能直覺比對最近兩天的血壓狀況（通常一天兩次），歷史紀錄由原來的「最近三天」改為**「最近四筆」**，直接顯示兩次平均值，並使用太陽與月亮圖示進行排版。

**排版範例：**
```text
[4 Catatan Terakhir]
🌙 6/29 20:15 (Malam)
   Rata-rata: 117 / 59 / 69
   ✅ Normal (Diastolik agak rendah)
───
☀️ 6/29 08:30 (Pagi)
   Rata-rata: 108 / 60 / 69
   ✅ Normal (Agak rendah)
```

## 4. 防重複登載機制 (寫入但提醒)

如果 Susi 因網路延遲或手誤連續傳送了兩次一模一樣的數據，系統**依然會將其寫入試算表**，但在回覆的訊息中會特別加上醒目的警告標誌以提醒檢查：
*   印尼文提示：`⚠️ (Data ini sama dengan sebelumnya / 此資料與上一筆相同)`
*   中文提示：`⚠️ (此筆資料與上一筆完全相同)`

## 5. 格式錯誤防噪機制 (無關訊息已讀不回)

為了避免機器人在群組中因為日常聊天或 Susi 傳送禮貌用語（如 `Terima kasih.`、`hi`、`ok`）而頻繁發送「格式錯誤」的警告洗版，系統實作了智慧防噪過濾：

### 🔍 判定邏輯
1.  **判定是否疑似血壓數據**：訊息必須**同時包含數字（0-9）與斜線（/）**。
2.  **處理與回覆對照**：
    *   **情境 A：符合格式** (如 `128/65/75 | 123/63/73`) $\rightarrow$ 成功寫入 Sheet，並回傳最近四筆歷史紀錄。
    *   **情境 B：疑似輸入血壓但格式打錯** (如 `128/65`，有數字與斜線但少了一組數據) $\rightarrow$ 回傳印尼文格式錯誤示範 `Format salah. Kirim 2 set...` 以引導修正。
    *   **情境 C：無關訊息與日常聊天** (如 `Terima kasih`、`hi`、`ok`) $\rightarrow$ 機器人**已讀不回（保持沉默）**，確保不干擾群組。

## 6. Google Apps Script 部署與 LINE 設定指南

當程式碼有更新時，請遵循以下步驟在網頁端進行設定與部署：

### 🔑 1. 設定 Token 指令碼屬性
1.  開啟 Google Apps Script 編輯器。
2.  點擊左側的 **「專案設定」** (齒輪圖示)。
3.  滾動至底部的 **「指令碼屬性」**，點選 **「新增指令碼屬性」**。
4.  屬性名稱填入：`LINE_CHANNEL_ACCESS_TOKEN`。
5.  屬性值填入：您的 LINE Channel Access Token。

**選填：危險血壓與系統告警推播 (`ALERT_LINE_USER_ID`)**

新增指令碼屬性 `ALERT_LINE_USER_ID`，填入兒子本人的 LINE userId。設定後：
- 血壓判定為 🔴 危險等級（極高危險／明顯偏高／明顯偏低）時，會用 LINE Push API 主動推播一則固定格式的警示訊息（等級＋數值＋時間）給這個 userId，跟 Susi 的對話完全分開，半夜也不會漏看。
- `writeToSupabase()` 寫入失敗時，同樣會推播一則系統告警（不含原始錯誤細節，只有固定分類文字），完整錯誤仍要看 Apps Script 執行紀錄。

**未設定這個屬性時**：兩種推播都會被安靜略過（只留 `console.warn`），不影響 Susi 這邊正常收到血壓回覆的主流程。取得自己的 userId 的方法：暫時把自己的 LINE 帳號加進 bot 對話，用 `console.log` 印出 `event.source.userId` 後移除，或參考 `bahasa-tw-bot` 用 `wrangler tail` 類似的 log 排查方式（GAS 這邊改看「執行項目」的 log）。

### 🚀 2. 更新部署 (不變更 Webhook URL)
若每次都選「新部署」會產生不同的 URL，需要去 LINE 後台重新設定。請使用以下步驟固定 URL：
1.  點擊右上角 **部署 (Deploy)** > **管理部署 (Manage deployments)**。
2.  在彈出的視窗中，點擊現用部署旁的 **鉛筆圖示 (編輯)**。
3.  在 **版本 (Version)** 下拉選單中，選擇 **「新版本 (New version)」**。
4.  點擊 **部署** 即可套用最新程式碼，且 Webhook URL 將維持不變。

### 👥 3. 允許 Bot 加入 LINE 群組
若要將 Bot 邀請入群且防止其自動退出：
1.  登入 [LINE Developers Console](https://developers.line.biz/console/)。
2.  進入您的 Channel，點選 **Messaging API** 標籤頁。
3.  滾動找到 **LINE Official Account features**。
4.  在 **Allow bot to join groups and multi-person chats** 旁點擊 **Edit**。
5.  在開啟的 Official Account Manager 中，將其設定為 **「Enabled (啟用)」**。
6.  設定完成後，將 Bot 從群組移出並重新邀請一次即可。

## 7. 設計決策 (Design Decisions)

### 📌 決策一：為什麼保留試算表中的「早/晚」獨立欄位？
雖然第一欄有詳細的時間戳記（例如 `6/29 16:34`），但我們依然保留了獨立的「早/晚」欄位：
1.  **醫學統計意義**：在心血管與高齡醫學中，「清晨高血壓」與「晚間高血壓」所代表的臨床指標（例如中風與動脈瘤破裂風險）不同。
2.  **便於圖表分析**：擁有獨立的早晚欄位，日後在 Excel 或 Google 試算表中做**篩選、樞紐分析表（Pivot Table）或繪製早晚趨勢圖**時，會非常直觀且快速。

### 📌 決策二：為什麼應該在「寫入時」判定早晚，而不是「讀出時」判定？
1.  **簡化系統架構**：Susi 均為當下即時登載。在寫入的當下，時段是確定且唯一的，一次判斷並永久存入試算表，避免後續讀取時還需要重複解析。
2.  **降低系統運算負擔**：如果在每次讀取最近四筆時都要重新解析日期字串、轉換時區並進行小時計算，會增加不必要的 Apps Script API 運算開銷，並增加潛在的時區判定錯誤風險。
